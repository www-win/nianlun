## 一句话概括

> **把散落在微信里的一次次「荐股」，让 AI 抽出来、整理成一个能反查的「荐股数据库」。**

客户和一堆首席在微信里聊股票，聊天记录里散着大量「谁推了什么票、为什么、看几倍」。这个模块就是把这些信息结构化、存档、可反查。

---

## 一、每支票沉淀三层信息 = 给每支股票建一张卡片

「沉淀」= 把散的信息攒起来存成档案。每支票一张卡片，从浅到深三层。以文档例子**江化微**为例：

### 第一层｜票的基本盘 —— 一句话档案

| 字段 | 含义 | 例子 |
|---|---|---|
| 股票名称 | 哪支票 | 江化微 |
| 谁推荐 | 哪个首席/朋友推的 | 张三首席 |
| 何时推荐 | 什么时候说的 | 2026年3月 |
| 当前现价 | 现在多少钱（数据库取，不求实时） | 40 元 |
| 目标市值 | 看到多少亿（全公司价值） | 500 亿 |
| 涨幅倍数 | 能涨几倍 | 看 2 倍 |
| 预计到目标时间 | 大概多久涨到 | 1 年内 |

### 第二层｜推荐逻辑 —— 凭什么值得买（分条列）

> - 逻辑 1：市值 400 亿 → 800 亿，有翻倍空间
> - 逻辑 2：**MOC 涨价**（产品涨价 → 公司多赚钱 → 股价涨）
> - 逻辑 3：……

### 第三层｜公司信息 —— 背景资料 + 谁说了啥（逐条列）

> - 公司做半导体材料，行业地位……
> - 张三说：「这票是核心标的」
> - 李四说：「短期涨太多，等回调」

**三层合起来 = 一支票的完整档案：身份 → 逻辑 → 众人评价。**

## 二、两个交叉视图 = 同一批数据，换两个角度看

就像通讯录能「按姓名找」也能「按公司找」——同一批荐股记录，两个入口：

|  | **视图 A：以票查人** | **视图 B：以人查票** |
|---|---|---|
| 点开的是 | 一支股票（如江化微） | 一个人（如张三首席） |
| 想知道 | 它**被谁推过、多少人推**、看几倍 | 他**推过哪些票**、看几倍 |
| 用途 | 找**核心标的**（推的人越多越重要） | 看某人的**战绩 / 风格** |
| 附加 | 可按涨幅倍数 / 推荐时间排序 | —— |

> 💡 **「有多少人在推」是视图 A 的关键**：推的人越多 = 越是**核心标的**（如客户说的「50 个核心标的」），要优先关注。
>
> 本质：**一份荐股记录（谁 + 哪票 + 何时 + 几倍），按「票」聚合就是视图 A，按「人」聚合就是视图 B。一份数据，两个入口。**

---

## 三、数据从哪来（决定技术怎么做）

| 数据 | 来源 | 说明 |
|---|---|---|
| 谁推了什么票 / 逻辑 / 目标价 | **AI 从聊天记录抽取** | 聊天记录是**唯一**原始数据源 |
| 现价（一个字段） | 「咱们的数据库」取数 | 单独来源，**不求实时** |

---

## 四、全流程一个完整故事

```
1. 微信里张三说：
   「江化微可以重点看，现在40块，能到500亿，MOC涨价逻辑，看2倍」

2. AI 抽成一条结构化记录：
   { 票:江化微, 人:张三, 时间:3月, 目标:500亿, 倍数:2倍, 逻辑:MOC涨价 }
   → 现价40元 另从数据库补上

3. 存进「江化微」的三层档案

4. 客户反查：
   · 点「江化微」→ 发现30人在推 → 这是核心标的！(视图A)
   · 点「张三」→ 看他今年推的20支票、战绩如何 (视图B)
```

**这就是整个模块在做的事：把聊天里的荐股，变成能反查的结构化档案。**

## 五、对接时围绕这 3 个问题问，就显得很懂

1. 一支票的**三层信息**（基本盘 / 逻辑 / 公司信息）字段够不够、还要加吗？
2. **以票查人** + **以人查票** 这两个视图，是不是你要的？
3. **现价**具体从哪个数据库 / 接口取？（正式稿「待确认清单」第 3 条，尚未定）
