## 三、数据导入 / 更新方案（沟通结论）

| 议题 | 结论 |
|---|---|
| 能否后台自动实时采集微信？ | **不可行**。微信在手机端无法做到后台实时采集。 |
| 正确路径 | 聊天记录先同步到电脑（微信 PC 版 / 电脑助手），再从电脑导出 → 导入本工具。 |
| 用户电脑空间不足（60G 已满） | 改用其闲置的 **Surface**，腾空后专门用于同步。 |
| 具体做法 | Surface 保持常开，微信 PC 版持续同步**新消息**；**历史记录从手机导出一遍**，两者合并即为完整数据。 |
| 增量导入 | 期望每次仅导入**新增**部分（而非每次全量约 4MB）。**待验证**：Windows 导出工具能否只导出「一天 / 一段时间」而非全量。 |
| 导入方式 | 目前仍为**人工手动导入**（暂不追求全自动）。 |

> **结论**：只要「增量导出 / 导入」链路能跑通，整体流程即打通，后续可持续迭代。

## 四、核心功能：好友分类（AI 自动分类）

先由 AI 为每位好友**定位分类**，不同类别走不同的分析维度：

1. **金融 / 投资类**（细分一级、二级）
   - **一级**（未上市投资人）：梳理其投资方向即可。
   - **二级**（股票）：**专门的「二级市场」模块，为最核心需求**（详见第五节）。
2. **政府类**：重点看其**职能**，要求**全、准**；含历史沟通内容、情绪等分析维度。
3. **产业类**：产业资源，如所任职公司、董事长身份等。
4. **其他**：亲人、朋友、家属等，归为一类，**不做专业分析**。

## 五、二级市场「股票模块」（客户描述最详细的部分）

### 每支票沉淀三层信息

- **第一层｜票的基本盘**：公司 / 股票名称、**谁推荐、何时推荐**、当前现价、目标市值（看到多少亿，如 500 亿 / 1000 亿）、**预计涨幅倍数**、预计涨到目标的大致时间。
  - *示例：江化微*
- **第二层｜推荐逻辑**：该票值得推荐的核心逻辑，分条列出（逻辑 1 / 2 / 3）。
  - *示例：和林微纳、国瓷材料，市值 400 亿 → 800 亿，逻辑为 MOC 涨价等*
- **第三层｜公司信息**：股票背后公司的相关信息，以及「谁说了什么」，逐条列出。

### 两个交叉视图（把信息串起来）

- **以「票」为维度**：反查某支票 —— 被谁推荐过、时间、价格、看多少倍、**有多少人在推**（推荐人数多 = 核心标的，如 50 个核心标的）。
  - 支持**排序**：按涨幅倍数、按推荐时间远近。
- **以「人 / 首席」为维度**：以人为单位，查看某人向其推荐过的所有股票。

### 数据来源

- **股票信息**：由 AI 从**聊天记录**中抽取（谁推了什么票、逻辑、目标价等）—— 聊天记录为唯一原始数据源。
- **现价数据**：从**「咱们的数据库」取数**，不追求实时。

### 📖 大白话解读（对接时看这段，把上面的黑话翻译成人话）

> 客户上面是用「投资圈黑话」描述的，这段把它翻译成大白话 + 一个完整例子，方便对接时对齐。
> 一句话概括：**把散落在微信里的一次次「荐股」，让 AI 抽出来、整理成能反查的「荐股数据库」。**

#### ①「每支票沉淀三层信息」= 给每支股票建一张卡片，从浅到深三层

以文档里的例子**江化微**为例：

- **第一层｜票的基本盘 = 一句话档案**：哪支票、**谁**推的、**何时**推的、现价、目标市值（看多少亿）、涨幅倍数（看几倍）、预计多久到。
  - 例：`江化微 · 张三首席推 · 3月 · 现价40元 · 目标500亿 · 看2倍 · 1年内`
- **第二层｜推荐逻辑 = 凭什么值得买**（分条列）：
  - 例：逻辑1 市值 400亿→800亿有翻倍空间；逻辑2 **MOC 涨价**（产品涨价→公司多赚→股价涨）；逻辑3……
- **第三层｜公司信息 = 背景资料 + 谁说了啥**（逐条列）：公司是做什么的、行业地位，以及每个人对它的评价（「张三说是核心标的」「李四说涨太多了等回调」）。

> 三层合起来 = 一支票的完整档案：**身份 → 逻辑 → 众人评价。**

#### ②「两个交叉视图」= 同一批荐股记录，换两个角度看（像通讯录能按姓名/按公司两种方式找）

|  | **视图A：以票查人** | **视图B：以人查票** |
|---|---|---|
| 点开的是 | 一支股票（如江化微） | 一个人（如张三首席） |
| 想知道 | 它**被谁推过、多少人推**、看几倍 | 他**推过哪些票**、看几倍 |
| 用途 | 找**核心标的**（推的人越多越重要） | 看某人的**战绩 / 风格** |
| 附加 | 可按涨幅倍数 / 推荐时间排序 | —— |

> 💡「**有多少人在推**」是视图A的关键：推的人越多 = 越是**核心标的**（如客户说的「50 个核心标的」），要优先关注。
> 本质：**一份荐股记录（谁+哪票+何时+几倍），按「票」聚合是视图A，按「人」聚合是视图B。**

#### ③ 数据从哪来（决定技术怎么做）

- 「谁推了什么票 / 逻辑 / 目标价」→ **AI 读聊天记录自动抽取**（聊天记录是唯一原始数据源）。
- 「现价」这**一个字段**→ 单独从「咱们的数据库」查（聊天里不一定有准确现价），不求实时。

#### 全流程一个完整故事

```
1. 微信里张三说：「江化微可以重点看，现在40块，能到500亿，MOC涨价逻辑，看2倍」
2. AI 抽成一条结构化记录：{票:江化微, 人:张三, 时间:3月, 目标:500亿, 倍数:2倍, 逻辑:MOC涨价}
   → 现价40元 另从数据库补上
3. 存进「江化微」的三层档案
4. 客户反查：点「江化微」→ 发现30人在推 → 这是核心标的！(视图A)
            点「张三」→ 看他今年推的20支票、战绩如何 (视图B)
```

## 五之补充、存储与数据流设计（落地端：微信小程序）

> 本节为技术方案落定，回答「导入的数据存哪、二级分析怎么直接用、要不要重导」。

### 落地端决策

- **确定用微信小程序**（最终用户 / 实习生现场用手机操作更顺手）。
- 小程序本地存储有**硬限制**：单键 ≤ 1MB、整体 ≤ 10MB。**全年聊天原文（几十 MB）存不下**，且与「原文不落盘」的隐私底线一致。

### 核心决策：本机留存「全年原文」作为桥梁 → 将来抽「荐股结果」

客户全年数据约 4MB，可容纳进小程序本地存储（10MB 内）。因此分两步走：

> **① 现在（已实现 2026-07-03）**：导入时把**原始聊天文本留存到本机**（分块绕过单键 1MB 限制），
> 让将来开发荐股分析时可**直接读回、无需客户重新导入**。仅存本机、绝不上传。
> **② 将来**：荐股分析读回原文 → 抽取**结构化荐股记录** → 持久化结果，供二级视图直接用。

> 隐私说明：此举对该客户放开「原文不落盘」（客户自有工具、明确要复用），数据仍仅存本机。

```
【导入时·现在已实现】原文进内存
   ├─► ① 算好友统计 + 报告（现有）
   ├─► ② 抽 30 条有界样本（现有，供 AI 画像/情绪）
   └─► ③ 【新增·已实现】把原始聊天文本留存到本机（分块绕过 1MB 单键限制）

【将来·荐股分析开发时】读回留存的原文 → AI 抽取荐股结果 → 持久化结果 → 二级视图直接用
   （无需客户重新导入，因为原文今天导入时已留存）
```

### 持久化清单（小程序本地存储）

| 存储内容 | 说明 | 体积 |
|---|---|---|
| 好友统计 / 报告 | 现有 | 小 |
| 每人 ≤30 条样本 | 现有，供 AI 画像/情绪 | 小 |
| **全年聊天原文（新增·已实现 2026-07-03）** | 分块留存，供将来荐股分析读回 | ~4MB（在 10MB 内） |
| 荐股结果（将来） | 结构化记录，供二级分析直接用 | 小（几百 KB 级） |

### 已知代价（对接时如实告知客户）

- **今天导入（含原文留存）后**，将来开荐股分析**不用客户重导**——原文已留在本机。✅
- 仍提醒客户：**务必保留导出文件**（微信/电脑里）作为兜底，以防本机数据被清。
- 单机 10MB 上限：4MB 原文 + 现有数据仍在余量内；若数据远超预期而超限，原文留存会告警但**不影响导入本身**（已做容错）。

### 若将来数据量远超 10MB（多年/多人）

则需改用**网页版 / 桌面版**（IndexedDB / 本地文件，存储无 10MB 限制）。当前单客户约 4MB，小程序方案足够。
